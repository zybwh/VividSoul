# VividSoul 对话气泡实施计划

## 文档信息

- 项目：`VividSoul`
- 范围：角色对话气泡、打字机文本、动作联动测试文案
- 关联文档：`../prd.md`、`../architecture.md`、`./implementation-plan.md`
- 日期：`2026-04-05`
- 状态：`implemented-first-pass`

## 1. 决策摘要

本计划固定以下产品与实现决策：

1. 当前先做`角色说话气泡`，不引入完整聊天面板或输入框。
2. 对话展示挂接到现有运行时 HUD，不新建独立 UI 框架。
3. 气泡必须支持`视觉气泡感`、`打字机效果`、`自动换行`、`自动关闭`和`复用`。
4. 当前先用现有 7 个内置动作作为触发源，为每个动作配置一句测试台词。
5. 后续真正接入 AI 时，沿用同一套气泡播放通道，不推翻本轮实现。

一句话结论：

> 这一轮不是做“聊天系统”，而是先做一条稳定、低分配、可复用的角色发言呈现链路。

## 2. 目标与非目标

### 2.1 目标

- 让角色在动作切换时能以`气泡`而不是纯文本框的形式说话。
- 让文本逐字出现，形成可感知的“说话中”反馈。
- 让气泡宽高根据文本自适应，并在达到最大宽度后自动换行。
- 让气泡在文本显示完成后自动关闭，且重复播放时不产生明显垃圾数据。
- 为现有 7 个动作提供固定测试文案，验证整条链路。

### 2.2 非目标

- 不做用户输入框。
- 不做多气泡历史记录。
- 不做完整会话管理、消息列表或聊天日志。
- 不做 TTS、口型同步和后端 AI 接口。
- 不做表情系统重构，只处理文本展示与动作联动。

## 3. 当前实现与差距

当前运行时已经具备：

- `DesktopPetRuntimeHud`：统一的 HUD、Canvas 和运行时 UI 入口。
- `DesktopPetRuntimeController`：统一的运行时控制器，负责姿势切换和菜单命令。
- `BuiltInPoseOptions`：现成的 7 个内置动作定义，可作为第一批触发源。
- `DesktopPetBoundsService`：现有屏幕空间边界计算思路，可复用于气泡定位。

当前主要差距：

- 没有角色对话 UI 容器。
- 没有逐字播放能力。
- 没有基于文本测量的气泡尺寸控制。
- 没有“显示一段时间后自动隐藏”的统一生命周期管理。
- 没有动作切换到台词播放的接线点。

## 4. 用户体验设计

### 4.1 展示位置

- 气泡显示在角色头顶上方。
- 位置由角色当前屏幕包围盒计算得出，而不是写死固定像素坐标。
- 当角色靠近屏幕边缘时，气泡位置需要向可视区域内收敛，避免超出屏幕。

### 4.2 视觉样式

气泡必须满足“看起来像在说话”而不是“纯色提示框”：

- 圆角主气泡背景。
- 单独的小尾巴，朝向角色头部区域。
- 柔和阴影或描边，避免直接贴在背景上显得廉价。
- 半透明浅色底配深色字，优先保证可读性。

建议先使用运行时代码生成的基础 UGUI 结构：

- `BubbleRoot`
- `BubbleShadow`
- `BubbleBody`
- `BubbleTail`
- `BubbleText`

这样本轮不依赖额外 prefab 和美术资源，先把交互链路跑通。

### 4.3 文本表现

- 文本以打字机形式逐字显示。
- 完整文本先被设置到文本组件中，再通过可见字符数控制显示进度。
- 超过最大宽度后自动换行。
- 默认不显示富文本解析，避免测试文案引入额外格式复杂度。

### 4.4 生命周期

单条消息的生命周期为：

1. 收到播放请求。
2. 取消并清空上一条播放状态。
3. 测量文本尺寸并设置气泡大小。
4. 逐字播放文本。
5. 播放完成后保持若干秒。
6. 淡出或直接隐藏。
7. 进入可复用空状态，等待下一次播放。

## 5. 技术方案

### 5.1 模块划分

建议在 `VividSoul/Assets/App/Runtime/App/` 下新增或扩展以下模块：

- `DesktopPetSpeechBubblePresenter`
- `SpeechBubbleDialogueCatalog`
- `SpeechBubbleDialogueLine`

职责划分：

- `DesktopPetRuntimeHud`
  - 管理 HUD 生命周期。
  - 持有并驱动气泡 presenter。
  - 订阅运行时的动作切换通知。
- `DesktopPetSpeechBubblePresenter`
  - 创建和缓存单个气泡视图。
  - 负责文本测量、定位、打字机、自动关闭和复位。
- `SpeechBubbleDialogueCatalog`
  - 管理 `poseId -> line` 的测试台词映射。

### 5.2 视图结构

气泡先采用单实例复用，不做对象池数组。

建议字段：

- `CanvasGroup`
- `RectTransform root`
- `RectTransform body`
- `RectTransform tail`
- `TMP_Text text`

原因：

- 当前同屏只需要一条活动发言。
- 单实例复用能满足“不要产生一堆 garbage data”的要求。
- 后续如果要做多角色或多消息，再扩展为池化结构更自然。

### 5.3 文本测量与最大宽度

建议使用 `TextMeshProUGUI`，并固定以下策略：

- 设定最大文本宽度，例如 `400f`。
- 在正式显示前，用文本组件测量最终所需尺寸。
- 气泡尺寸 = 文本首选尺寸 + 内边距。
- 文本组件开启自动换行，禁止水平无限扩展。

目标效果：

- 短文本显示为较窄气泡。
- 长文本在达到阈值后自动换成多行。
- 气泡高度随行数增加而增长。

### 5.4 打字机效果实现

建议实现方式：

- 一次性设置完整文本到 `TMP_Text.text`。
- 通过 `maxVisibleCharacters` 逐步增加可见字符数量。
- 播放期间使用协程或统一 `Update` 驱动均可，但必须支持中途取消。

关键要求：

- 新消息到来时，旧播放必须立即取消。
- 取消后要立刻把旧文本不可见，避免用户看见残留。
- 打字速度先做成常量，后续再开放为配置。

### 5.5 自动关闭与复用

复用策略：

- `Awake` 或首次使用时创建一次气泡。
- 后续只做 `Show` / `Hide` / `Reset`。
- 不在每次发言时 `Instantiate` 或 `Destroy`。

重置动作至少包括：

- 停止当前协程或取消当前播放令牌。
- 隐藏根节点。
- 清空文本内容。
- 将 `maxVisibleCharacters` 置零。
- 恢复透明度、尺寸和内部定时器状态。

### 5.6 动作联动接线

本轮优先接到现有 7 个内置动作：

- `vrma_01`
- `vrma_02`
- `vrma_03`
- `vrma_04`
- `vrma_05`
- `vrma_06`
- `vrma_07`

建议接线方式：

1. 在 `DesktopPetRuntimeController` 中为内置动作播放增加通知事件。
2. 事件携带 `poseId` 或动作标识。
3. `DesktopPetRuntimeHud` 订阅该事件。
4. HUD 从 `SpeechBubbleDialogueCatalog` 取出台词并交给 presenter 播放。

这样做的好处：

- UI 不直接依赖姿势菜单实现。
- 后续无论动作来自菜单、AI 指令还是别的输入源，都能复用同一展示通道。

## 6. 测试台词

当前为 7 个内置动作固定以下文案：

- `vrma_01`：`先让我正式亮个相吧。`
- `vrma_02`：`嗨，今天也请多关照。`
- `vrma_03`：`耶，这个手势是不是很有精神？`
- `vrma_04`：`目标锁定，准备发射可爱光波。`
- `vrma_05`：`转一圈，看看我今天的状态。`
- `vrma_06`：`这个角度不错，拍照一定很上镜。`
- `vrma_07`：`先蹲一下，我要开始认真了。`

## 7. 分阶段实施计划

### Phase 1：动作触发与文案目录

目标：

- 先把“动作切换会触发一句话”这条逻辑链打通。

任务：

- 定义动作发言事件。
- 定义 7 个动作到文案的映射表。
- 在内置动作播放入口触发通知。

完成标准：

- 每次切换到 7 个内置动作之一，HUD 都能收到对应 `poseId`。

### Phase 2：气泡视图与基础展示

目标：

- 在 HUD 上生成一个真正像气泡的 UI。

任务：

- 创建单实例气泡根节点。
- 添加背景、尾巴、文本和可选阴影。
- 实现基础显示与隐藏。

完成标准：

- 能在角色上方显示一个视觉上明确的对话气泡。

### Phase 3：文本测量与自适应布局

目标：

- 让气泡宽高跟随文本内容变化。

任务：

- 接入 `TextMeshProUGUI`。
- 添加最大宽度限制。
- 根据测量结果设置气泡尺寸。
- 验证短句和长句的换行行为。

完成标准：

- 长句会自动换行，气泡不会无限横向拉长。

### Phase 4：打字机与自动关闭

目标：

- 让发言具备节奏感和完整生命周期。

任务：

- 实现逐字播放。
- 实现播放完成后的停留时长。
- 实现自动关闭。
- 实现中途新消息覆盖旧消息。

完成标准：

- 文本会逐字出现，结束后自动关闭。

### Phase 5：复用与稳定性清理

目标：

- 保证连续触发时没有明显残影和无谓分配。

任务：

- 清理播放取消逻辑。
- 实现统一 `Reset`。
- 确保重复播放不新增 UI 实例。
- 检查模型切换或 HUD 关闭时的清理行为。

完成标准：

- 连续切动作时不会看到旧文本残留。
- 运行期间不重复创建和销毁气泡对象。

## 8. 任务拆分建议

### Task Group A：运行时事件接线

- 内置动作播放通知事件
- HUD 订阅和解绑
- 文案目录查询

### Task Group B：气泡视图构建

- Canvas 子节点结构
- 背景和尾巴绘制
- 文本组件挂接

### Task Group C：文本播放控制

- 文本测量
- 打字机播放
- 自动关闭计时
- 中途取消与覆盖

### Task Group D：定位与边界

- 角色头顶定位
- 屏幕边界收敛
- 模型缺失时的安全回退

## 9. 验收标准

满足以下条件即可认为本计划完成：

1. 当前 7 个内置动作都能触发一条测试台词。
2. 说话 UI 呈现为气泡，而不是普通纯文本提示。
3. 文本按打字机方式逐字出现。
4. 气泡宽高会根据文本长度自适应变化。
5. 文本达到最大宽度后会自动换行。
6. 文本播放完成后气泡会自动关闭。
7. 连续触发动作时，旧内容会在下次显示前被彻底清空。
8. 整个过程复用同一个气泡实例，不反复创建销毁。

## 10. 风险与控制

### 风险 1：UGUI 气泡样式太像提示框

控制：

- 必须包含尾巴和圆角主体两个可识别特征。
- 通过阴影或描边增强“气泡感”。

### 风险 2：逐字播放产生多余 GC

控制：

- 使用 `TextMeshProUGUI.maxVisibleCharacters`，不逐帧拼接字符串。

### 风险 3：动作切换过快导致残影

控制：

- 每次新消息前统一执行取消和重置。
- 旧文本在新播放开始前必须先被隐藏。

### 风险 4：气泡位置在边缘溢出屏幕

控制：

- 显示前做屏幕边界夹取。
- 尾巴朝向角色，但气泡主体允许左右偏移。

## 11. 当前建议的实施顺序

推荐严格按下面顺序推进：

1. `Phase 1：动作触发与文案目录`
2. `Phase 2：气泡视图与基础展示`
3. `Phase 3：文本测量与自适应布局`
4. `Phase 4：打字机与自动关闭`
5. `Phase 5：复用与稳定性清理`

原因：

- 先打通动作到发言事件，后面的 UI 和播放逻辑才能稳定接入。
- 先有静态可见气泡，再做尺寸自适应和打字机，调试效率最高。
- 复用和清理放在最后集中收口，更容易发现真实残留问题。

## 12. 实施开始时的明确决定

当前可以直接执行的工程决定如下：

- UI 入口：`DesktopPetRuntimeHud`
- 第一触发源：现有 7 个内置动作
- 文本组件：`TextMeshProUGUI`
- 复用策略：单实例气泡复用
- 首轮目标：只做展示链路，不接 AI、不接输入框

这意味着下一步实现不应先去做：

- 多消息队列
- 聊天记录面板
- 网络对话协议
- TTS 或口型同步

而应该先把“动作触发一句话并稳定显示/关闭”做完整。

## 14. 与原计划的实现差异

实际落地时，与最初方案有两点重要差异：

1. 文本组件没有继续使用 `TextMeshProUGUI`，而是改为 `UGUI Text + OS 动态字体`。
2. 原因是已构建 player 在运行时动态创建 TMP 组件时，缺少可用的 TMP 默认设置资源，会触发空引用；当前实现优先保证 shipped player 稳定。

视觉实现上也从“单纯圆角气泡”扩展为运行时拼装的多层结构：

- `BodyShadow`
- `BodyDepth`
- `Body`
- `Highlight`
- `TailLarge / TailSmall / TailTiny`

这意味着后续继续调风格时，不需要推翻播放逻辑，只需要继续调尺寸、颜色、层次和尾泡布局。
