# VividSoul 运行时交互壳层重构计划

## 文档信息

- 项目：`VividSoul`
- 范围：运行时入口、右键菜单、聊天入口、角色库入口、设置入口、状态反馈
- 关联文档：`../prd.md`、`../architecture.md`、`./model-library-import-plan.md`
- 日期：`2026-04-07`
- 状态：`draft`

## 1. 决策摘要

本计划先固定以下方向，用于后续 UI 重构讨论与实施：

1. 当前问题不是单个控件 bug，而是运行时交互壳层的信息架构已经失衡。
2. 不引入永久可见的独立悬浮主入口，避免对桌面常驻体验造成持续干扰。
3. `右键菜单` 保留为唯一导航入口，但必须回退为一级命令菜单，不再承载持续增长的内容浏览。
4. `角色库` 与 `聊天` 属于一等功能，应统一进入同一个主面板容器，而不是继续拆散在菜单、按钮和设置附属区中。
5. `动作` 不再作为主导航能力暴露，而是降级为受控能力层：由 `LLM` 或运行时逻辑选择预定义动作，`Unity` 负责校验和执行。
6. 手动动作入口只保留低频的 `动作管理 / 动作注册` 页面，用于配置、映射、测试和调试，不再保留当前面向用户的动作按钮或动作主页面。
7. `设置` 只负责配置，不再混入角色管理这类高频内容操作。
8. `状态提示` 与 `对话气泡` 只承担反馈职责，不承担导航职责。
9. `最近使用` 可以保留，但只能作为辅助能力，不能再伪装成完整角色库。

一句话结论：

> 先把 `VividSoul` 的运行时 UI 从“堆入口”重构成“分层壳层”，再分别优化角色库、聊天和动作体验。

## 2. 背景与问题定义

当前运行时入口主要分散在以下几个表面：

- 宠物本体点击与右键菜单
- 常驻聊天按钮
- 聊天面板
- 设置弹窗
- 状态消息与对话气泡

随着能力增长，已经出现以下结构性问题：

### 2.1 入口职责重叠

- `聊天` 同时存在按钮入口、右键入口、气泡反馈。
- `角色` 同时分散在 `添加角色`、右键菜单里的 `角色库`、设置里的 `角色库管理`。
- `动作` 既有右键姿势选择，也有气泡联动、随机动作逻辑，以及未来的 `LLM` 联动可能。

### 2.2 菜单承载了不适合菜单的内容

- `角色库` 会持续增长，不适合放在右键子菜单中。
- `姿势/动作` 即使保留手动入口，也不适合继续依赖级联菜单。
- 菜单更适合瞬时命令，不适合内容浏览、筛选、管理和状态感知。

### 2.3 配置与内容管理混在一起

- `设置` 中已经承担 `LLM 配置`。
- 同一个设置窗口又放进了 `角色库管理`。
- 这会让“配置系统”和“管理内容”边界变得模糊，也会让设置窗口持续膨胀。

### 2.4 反馈层正在被迫承担导航含义

- 对话气泡既是角色表达，又在一定程度上承接了对聊天系统的可见反馈。
- 底部状态消息既显示错误，也承担导入/删除确认反馈。
- 当导航入口缺位时，反馈层会被用户误认为功能入口的一部分。

### 2.5 当前显示与真实数据边界不一致

- 角色库的数据事实源已经是 `Content/Models/`。
- 但部分 UI 仍然基于“最近使用缓存”来展示角色。
- 这会直接制造“模型丢失了”的认知错误。

## 3. 目标与非目标

### 3.1 目标

- 为运行时 UI 建立清晰、稳定的分层结构。
- 明确每个入口的职责边界，减少重复入口。
- 让 `角色库`、`聊天`、`动作`、`设置` 成为可持续扩展的独立能力。
- 降低未来继续加功能时对右键菜单和设置弹窗的压力。
- 为后续视觉设计、状态设计和交互动画留出一致的落点。

### 3.2 非目标

- 本计划不直接定义最终视觉风格稿。
- 本计划不在本轮引入复杂拖拽布局或多窗口系统。
- 本计划不重做聊天后端、模型导入协议或内容格式。
- 本计划不要求一次性完成所有 UI 重构。

## 4. 设计原则

### 4.1 菜单适合命令，不适合内容

- 右键菜单只保留短小、低频、无需浏览上下文的命令。
- 会增长、需要滚动、需要状态感知的能力进入正式面板。

### 4.2 不新增永驻悬浮入口

- 桌宠是长期悬浮产品，不适合再叠加一个持续可见的主入口按钮。
- 新入口必须按需出现，不能成为常驻视觉噪音。

### 4.3 配置与内容分离

- `设置` 只承载偏静态的系统配置。
- `角色库`、`聊天` 作为独立工作区，不再塞进设置。
- `动作管理 / 注册` 作为低频管理能力保留，但不再作为用户主工作区。

### 4.4 主能力必须有稳定入口

- `角色库`、`聊天` 都是主能力。
- 主能力不能只藏在右键菜单二级层级中。
- 主能力应从右键菜单一级项进入统一主面板。
- `动作` 不作为主能力暴露，而作为内部能力层由聊天与运行时行为复用。

### 4.5 反馈只反馈

- 气泡、Toast、状态栏只用于表达结果与状态。
- 不依赖它们承接导航或主操作。

### 4.6 数据事实源必须与展示事实源一致

- `角色库 UI` 必须直接消费角色库事实源。
- `最近使用` 只能是排序、快捷入口或推荐来源，不能替代完整列表。

## 5. 目标交互结构

建议将运行时 UI 划分为四层：

### 5.1 直接交互层

由宠物本体承担，负责：

- 点击触发表达或聚焦
- 拖拽移动
- 基础 hover / focus 状态

这一层不负责复杂导航。

### 5.2 快捷命令层

保留轻量级线性右键菜单，只放一级命令：

- 打开聊天
- 打开角色库
- 打开设置
- 置顶 / 穿透 / 退出

建议移除：

- 角色库内容列表
- 多层角色切换子菜单
- 长列表姿势选择
- 常驻聊天按钮入口
- 面向用户的独立动作入口

### 5.3 主工作区层

将高频、成长型能力放进统一的主工作区面板。该面板由右键菜单的一级入口打开，首轮建议包括两个页签或两个一级视图：

- `聊天`
- `角色`

各自职责：

- `聊天`
  - 聊天记录
  - 输入框
  - 未读状态
  - 当前连接状态
- `角色`
  - 已安装角色库
  - 当前角色标记
  - 导入入口
  - 删除 / 应用 / 显示文件等操作

### 5.4 系统配置层

`设置` 独立存在，只承载：

- `LLM` 配置
- 常规系统设置
- `动作管理 / 注册`
- 运行参数
- 调试与诊断入口

不再承载：

- 角色库浏览
- 角色切换
- 高频内容管理

## 6. 建议入口方案

### 6.1 已确认方案

采用“单一线性右键菜单 + 统一主面板”的结构：

- 不新增常驻悬浮主入口
- 不占用左键打开主界面
- 保留右键菜单作为唯一导航入口
- 右键菜单取消二级菜单
- `聊天 / 角色库` 通过右键菜单一级项打开同一个主面板容器
- `设置` 仍为独立配置窗口或独立配置页，并承载低频的 `动作管理 / 注册`

这样可以解决：

- 桌宠长期悬浮时新增按钮带来的视觉干扰
- 左键已经承担互动职责，无法再安全复用为主工作区入口
- 角色库和动作列表继续增长后对子菜单的挤压
- 聊天、角色、设置分裂成多套入口的问题

### 6.2 菜单形态决策

当前倾向保留桌面端更稳妥的线性菜单，不采用圆盘式菜单：

- 线性菜单更适合文本标签
- 线性菜单更适合桌面鼠标扫读
- 圆盘菜单更适合少量、无文字、手势感强的即时操作
- 当前能力包含 `聊天 / 角色库 / 设置` 这类语义明确但不适合图标化猜测的入口

因此，本计划先固定：

- 继续使用线性右键菜单
- 不再扩展二级菜单
- 内容浏览统一进入面板

## 7. 当前界面到目标界面的映射

### 7.1 右键菜单

当前：

- 添加角色
- 角色库子菜单
- 姿势选择子菜单
- 聊天
- 设置
- 随机走动
- 应用示例移动行为
- 退出

目标：

- `聊天`
- `角色库`
- `设置`
- `系统`
- `退出`

说明：

- `聊天 / 角色库` 是主能力一级项
- 不再出现 `角色库 -> 模型列表` 这类二级菜单
- 不再出现 `姿势选择 -> 长列表` 这类二级菜单
- `动作` 不再出现在主导航中
- `系统` 可容纳少量低频命令，如置顶、穿透等

### 7.2 聊天按钮

当前：

- 独立常驻按钮
- 同时承担主入口与未读提示

目标：

- 取消独立常驻按钮
- 入口回收至右键菜单一级项
- 未读数字挂在主面板内部或右键菜单文案状态上

### 7.3 设置弹窗

当前：

- `LLM` 配置
- 常规设置
- 角色库管理

目标：

- 只保留配置
- 角色库迁出到统一主面板中的独立 `角色` 页签
- 增加低频 `动作管理 / 注册` 区，用于维护动作目录、映射关系和测试入口

### 7.4 状态消息与对话气泡

当前：

- 用户错误
- 操作成功
- 聊天相关反馈
- 角色表达

目标：

- 对话气泡：仅角色表达与剧情式反馈
- 状态消息：仅系统结果、失败、完成提示
- 不再承接“你应该从哪儿继续点”的职责

## 8. 实现落点建议

### 8.1 运行时壳层协调器

建议将 `DesktopPetRuntimeHud` 逐步收敛为壳层协调器，而不是继续直接堆叠各类 UI 创建逻辑。

建议引入统一的壳层状态概念，例如：

- 当前打开的主面板页签
- 是否显示设置
- 是否显示快捷命令菜单
- 是否存在未读聊天
- 是否存在系统状态消息

### 8.2 Presenter 职责拆分

建议逐步拆分为：

- `RuntimeShellPresenter`
- `ChatPanelPresenter`
- `CharacterLibraryPresenter`
- `SettingsPresenter`
- `ActionRegistryPresenter`
- `StatusPresenter`

现有模块的大致演进方向：

- `DesktopPetRuntimeHud`
  - 负责壳层协调
- `DesktopPetChatOverlayPresenter`
  - 从独立 overlay 演进为主工作区中的 `聊天` 页面
- `DesktopPetSettingsWindowPresenter`
  - 收缩为配置与动作管理窗口
- 角色库相关 UI
  - 从设置窗口中独立出来

### 8.3 数据消费边界

- `角色页` 直接消费角色库事实源
- `最近使用` 只作为排序或快捷入口
- `聊天页` 直接消费会话、状态事件与受控动作意图
- `状态层` 只消费可展示消息，不反向驱动导航

### 8.4 LLM 驱动动作协议

动作能力采用“预定义动作集合 + 结构化动作意图”的受控模型：

- `Unity` 维护 `ActionCatalog`
- `LLM` 只返回预定义动作 `id` 与少量受限参数
- `Unity` 通过统一 `ActionDispatcher` 校验后执行
- 执行失败时降级为纯文本反馈，不允许自由调用任意运行时方法

建议的职责拆分：

- `ActionCatalog`
  - 定义合法动作、参数约束、可见名称、触发条件
- `LlmActionIntent`
  - 表示一次结构化动作意图，例如 `play_pose("wave")`
- `ActionDispatcher`
  - 负责校验、路由与执行
- `ActionRegistryPresenter`
  - 负责手动注册、映射、启停、测试和调试

关键约束：

- 动作集合由 `Unity` 预定义，不由 `LLM` 自由生成
- `LLM` 只拥有“选择动作”的权限，不拥有“定义动作”的权限
- 不暴露任意方法名、任意文件操作、任意系统命令给 `LLM`
- 聊天是高频触发动作者，手动动作页面只是低频管理工具

## 9. 分阶段实施计划

### Phase 0：信息架构定稿

- 固定“线性右键菜单 + 统一主面板”结构
- 固定“不新增悬浮主入口、不复用左键打开主界面”约束
- 确认 `聊天 / 角色 / 设置 / 动作管理` 的边界

完成标准：

- 所有主能力各有唯一主入口
- 不再存在“同一能力散落三处入口”的结构

### Phase 1：角色与设置解耦

- 将 `角色库管理` 从设置窗口迁出
- 保留设置窗口中的纯配置项
- 建立独立 `角色` 面板骨架

完成标准：

- 设置页不再承载角色库列表
- 角色能力有独立容器

### Phase 2：聊天入口统一

- 将现有聊天 overlay 合并进主工作区
- 常驻聊天按钮降级为主入口或页签状态的一部分
- 未读提示与连接状态改挂在统一入口上

完成标准：

- 聊天只有一个主要打开方式
- 未读提示位置唯一且稳定

### Phase 3：动作能力层收敛

- 将动作从“用户主导航项”降级为“受控能力层”
- 设计第一版 `ActionCatalog`
- 为聊天链路增加结构化 `LlmActionIntent`
- 建立低频 `动作管理 / 注册` 页面替代当前动作按钮与测试入口

完成标准：

- `LLM` 只能选择预定义动作
- 用户主导航中不再存在独立动作按钮
- 手动动作能力只保留一个低频管理页面

### Phase 4：右键菜单瘦身

- 删除角色库长列表与姿势长列表
- 右键菜单回归命令型入口
- 删除面向用户的动作入口

完成标准：

- 右键菜单项数明显下降
- 无持续增长内容继续堆积在菜单中

### Phase 5：反馈层重整

- 区分角色表达反馈与系统状态反馈
- 统一成功、失败、确认、等待等状态文案样式
- 清理当前入口缺位导致的“反馈承担导航”问题

完成标准：

- 气泡与状态消息职责不再混淆
- 用户能稳定判断“这是反馈”还是“这是入口”

## 10. 风险与注意事项

### 风险 1：一次性重构过大

- UI 创建逻辑目前集中在少数类中
- 一次性全面替换风险高

缓解：

- 先定壳层结构，再按 `角色 -> 聊天 -> 菜单 -> 反馈` 顺序逐步迁移

### 风险 2：旧入口移除后可发现性下降

- 用户可能已经习惯某些现有按钮或右键项

缓解：

- 保留过渡期映射
- 在新入口上线初期增加一次性提示

### 风险 3：视觉方案先行会放大返工

- 若先做精细视觉，再改结构，返工成本高

缓解：

- 先完成信息架构和容器边界
- 视觉样式放在结构稳定后统一迭代

## 11. 待确认问题

1. `动作管理 / 注册` 放在设置的高级页里，还是保留为独立的低频工具页？
2. 设置是否继续保留弹窗形式，还是也纳入主工作区中的单独页签？
3. 是否需要在主工作区中引入更明显的当前角色卡片，作为聊天与角色管理之间的共享上下文？
4. 右键菜单中是否保留 `系统` 一级分组，还是直接平铺少量系统命令？
5. 第一版 `ActionCatalog` 只开放姿势类动作，还是同时开放移动、行为 preset 与表情类动作？

## 12. 验收标准

完成本计划后的目标状态应满足：

1. 用户能明确区分 `聊天`、`角色`、`设置` 与低频 `动作管理` 四类能力。
2. `角色库` 不再因为 UI 仍使用缓存列表而出现“模型消失”的认知问题。
3. 右键菜单不再承载长列表内容。
4. 主导航中不再存在面向用户的独立动作入口，动作由 `LLM` 或运行时逻辑通过预定义协议触发。
5. 设置窗口不再混入高频角色管理。
6. 状态消息和对话气泡只承担反馈职责。
