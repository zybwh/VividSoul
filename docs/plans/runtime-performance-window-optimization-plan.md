# VividSoul 运行时性能与小窗口优化计划

## 文档信息

- 项目：`VividSoul`
- 范围：运行时桌宠窗口形态、透明命中检测、常驻动画负载、模型切换资源回收、用户导入 `VRM` 的运行时性能档位
- 关联文档：`../STATUS.md`、`../architecture.md`、`./desktop-pet-runtime-ui-restructure-plan.md`
- 日期：`2026-04-08`
- 状态：`phase-1-in-progress`

## 1. 决策摘要

本计划固定以下方向，作为本轮性能治理与实现基线：

1. 当前运行时从“近似整屏透明覆盖层”转向“默认跟随宠物的小透明窗口”。
2. 小窗口仍必须保留：
   - 左键直接拖动宠物在桌面上的全局位置
   - 滚轮直接缩放宠物
   - 右键菜单作为主入口
3. `聊天 / 设置 / 角色库` 等需要更大空间的交互面板，不继续塞进始终固定的小窗口，而是在打开时临时切到扩展窗口态，关闭后恢复小窗口态。
4. 透明窗口不再继续依赖 `Opacity + ReadPixels` 的逐帧像素命中检测；优先改为 `Raycast` / 轻量命中代理路径。
5. 待机动作保留，但不再高频切换大动作；“轻微活着感”和“明显姿势切换”必须拆开处理。
6. 模型切换必须补齐正式资源回收链，避免多模型切换后内存只涨不回。
7. 用户导入模型仍允许保留原始文件，但运行时要支持“原始 / 均衡 / 省内存”性能档位，通过自定义加载策略降低贴图与运行时资源开销。

一句话结论：

> 当前这轮优化不是单点修补，而是把 `VividSoul` 从“全屏透明实时覆盖层”收束为“默认小窗桌宠 + 面板临时扩展 + 低成本待机 + 可控导入负载”的长期可用运行时形态。

## 2. 背景与问题定义

当前运行时的高常驻占用主要来自以下叠加因素：

- 透明窗口采用 `Opacity` 命中检测，底层在运行时通过 `ReadPixels` 判定鼠标下像素透明度，容易触发 GPU/CPU 同步。
- 桌宠窗口当前近似覆盖整块桌面显示区域，窗口与图形 backing surface 偏大。
- 运行时常驻 `60 FPS`，待机也持续绘制。
- `VRM` 模型启用了较激进的常驻动画策略，例如 `updateWhenOffscreen` 和 `AlwaysAnimate`。
- 待机动作系统除了轻微呼吸外，还较频繁地调度明显姿势切换。
- 模型切换后缺少正式资源回收链，导致多模型切换场景下内存容易居高不下。

这些问题叠加后，会让桌宠在“明明只是常驻待机”的情况下表现出偏高的 `CPU` 与内存占用。

## 3. 目标与非目标

### 3.1 目标

- 将默认运行态收束为围绕宠物本体的小透明窗口。
- 保留宠物最核心的桌面交互：左键拖动、滚轮缩放、右键菜单。
- 保留待机动作带来的存在感，但明显降低高频大动作切换。
- 降低待机常驻 `CPU` 与图形开销。
- 降低多次切换模型后的常驻内存。
- 为用户导入的大模型提供运行时性能档位，而不是只能被动接受原始资源体积。

### 3.2 非目标

- 本轮不做完整渲染管线重构。
- 本轮不做运行时网格简化或骨骼重拓扑。
- 本轮不修改用户导入的原始 `VRM` 文件内容。
- 本轮不重写聊天 / 设置 / 角色库功能本身，只调整其窗口承载方式。

## 4. 实现总览

本轮实现分为五个工作流：

1. 小透明窗口与扩展面板双态窗口。
2. 透明命中检测降载。
3. 待机动作与常驻动画策略降载。
4. 模型切换资源回收。
5. 用户导入 `VRM` 的运行时性能档位。

## 5. 方案设计

### 5.1 双态窗口

默认使用 `CompactPetWindow`：

- 小窗口尺寸由当前模型屏幕包围盒加边距计算得出。
- 模型默认在窗口内居中或脚底对齐。
- 拖动宠物时直接移动 OS 窗口位置，而不是继续依赖整屏窗口内的模型平移。
- 滚轮缩放后，模型缩放与窗口尺寸联动更新。

当打开 `聊天 / 设置 / 角色库` 时切到 `ExpandedPanelWindow`：

- 扩展窗口保留原先较大的 HUD/UI 展示能力。
- 关闭这些面板后恢复 `CompactPetWindow`。
- 不要求同时支持多个扩展窗口形态，本轮统一为一个扩展态即可。

### 5.2 命中检测

当前高成本路径：

- `TransparencyType = Alpha`
- `HitTestType = Opacity`
- 逐帧 `ReadPixels`

替换方向：

- 优先切到 `HitTestType = Raycast`
- 为模型补一个轻量运行时碰撞体作为宠物命中代理
- 继续保留 `GraphicRaycaster` 让 HUD / 菜单 / 设置 UI 在扩展态下正常命中

### 5.3 待机动作与动画策略

保留两层待机：

- 轻量持续层：呼吸、头部轻摆、极轻微 sway
- 低频事件层：偶发姿势切换

具体调整：

- 拉长随机姿势切换间隔
- 缩减待机控制器实际跟踪的骨骼集合
- 放宽 `SkinnedMeshRenderer` 与 `Animator` 的常驻更新策略
- 按交互状态动态调整帧率

### 5.4 模型切换资源回收

模型切换后执行：

1. 停止当前动画与过渡状态
2. `Destroy(previousModelRoot)`
3. 等待一帧
4. `Resources.UnloadUnusedAssets()`
5. `GC.Collect()`

这条链只在低频切换节点执行，不能放入常驻循环。

### 5.5 运行时导入性能档位

新增导入性能档位：

- `原始`
- `均衡`
- `省内存`

首轮能力：

- 通过自定义 `ITextureDeserializer` 在运行时导入时做贴图降采样
- `均衡` 档限制超大贴图到 `2048`
- `省内存` 档限制超大贴图到 `1024`
- 对模型体积、贴图尺寸、Renderer 数量做导入前体检与提示

## 6. 实施顺序

### 第一阶段

- 文档落地
- 替换命中检测路径
- 为模型补运行时碰撞代理
- 模型切换后加入资源回收链
- 调整待机动作频率
- 放宽常驻动画策略

### 第二阶段

- 实现 `CompactPetWindow`
- 实现 `ExpandedPanelWindow`
- 改造拖拽与缩放到窗口语义

### 第三阶段

- 增加 `VRM` 运行时性能档位
- 增加导入前性能体检与用户提示

## 7. 验收标准

- 待机常驻路径不再依赖 `ReadPixels` 命中检测。
- 默认状态下窗口明显小于整屏覆盖区域。
- 左键拖动可以直接移动宠物在桌面上的全局位置。
- 滚轮缩放后窗口尺寸会跟随调整，宠物位置稳定。
- 待机动作仍存在，但明显大动作切换频率降低。
- 连续切换多个模型后内存会回落，而不是一路只升不降。
- 用户导入大贴图模型时，`均衡 / 省内存` 档能明显降低运行时内存压力。
